# postgres-schema-merge

Merge data from one PostgreSQL schema into another while preserving foreign-key relationships. Primary keys are regenerated in the target schema and old → new id mappings are tracked, so FK columns get rewritten transparently.

> ⚠️ **DDL must be identical on both schemas.** This tool only copies rows and remaps foreign keys — it does **not** create or alter tables. Both the source and the target schema must already have matching column names, types, and constraints. If they differ, run your DDL migration first and then use this tool to move data.

## Why?

Common scenarios where this is useful:

- Consolidating multi-tenant databases that were originally provisioned as separate schemas.
- Merging two environments (e.g. staging into production) without conflicts on serial / identity primary keys.
- Importing a dump from a legacy schema into a new schema that shares the same DDL.

The tool does not generate DDL — both schemas must already have the same table structure. It only copies rows and remaps foreign keys.

## How it works

1. **`temp_migration`** — a single tracking table holds every `(source_schema, source_table, source_column, old_id, new_id)` mapping produced during the run.
2. **`get_table_hierarchy(source_schema)`** — topologically sorts the tables in a schema based on FK constraints, so children are migrated after their parents.
3. **`migrate_table(source_schema, target_schema, tbl)`** — copies a single table:
   - Detects the single-column primary key.
   - For each FK column, looks up `old_id → new_id` in `temp_migration` (handles both `uuid` and `numeric` keys).
   - Inserts into the target schema and records the new mappings.
4. **`migrate_schema(tables, source_schema, migration_schema)`** — iterates through a list of tables and calls `migrate_table` for each.

## Requirements

- PostgreSQL 12+ (uses `generated by default as identity`, `format()`, `array_agg(... ORDER BY ...)`).
- Both schemas must exist with **identical DDL** (matching column names, types, and constraints).
- Each migrated table needs a **single-column primary key**. Tables without one are skipped with a notice.

## Installation

Run the script once against the database you want to migrate within:

```bash
psql -d your_database -f merge.sql
```

This creates `public.temp_migration` and the helper functions.

## Usage

```sql
-- 1. Get tables in dependency order
SELECT get_table_hierarchy('legacy');

-- 2. Migrate everything from "legacy" into "public"
SELECT migrate_schema(
    get_table_hierarchy('legacy'),  -- tables in correct order
    'legacy',                        -- source schema
    'public'                         -- target (migration) schema
);

-- 3. Inspect the id mapping
SELECT * FROM public.temp_migration LIMIT 20;
```

You can also migrate a custom subset:

```sql
SELECT migrate_schema(
    ARRAY['users', 'orders', 'order_items'],
    'tenant_a',
    'public'
);
```

`source_schema` and `migration_schema` both default to `'public'`, but the function refuses to run when they are equal.

## Function reference

| Function | Purpose |
| --- | --- |
| `get_table_hierarchy(source_schema text DEFAULT 'public') → text[]` | Returns the schema's tables sorted by FK dependency (parents before children). Self-references are ignored. Cycles emit a warning and remaining tables are appended. |
| `migrate_table(source_schema text, target_schema text, tbl text) → text` | Copies one table and records id mappings. Returns the row count. |
| `migrate_schema(tables text[], source_schema text DEFAULT 'public', migration_schema text DEFAULT 'public') → text[]` | Migrates the listed tables in order. Raises if both schemas are the same. |

## Limitations

- Only single-column primary keys are supported.
- Composite foreign keys are not handled.
- FK columns are assumed to be either `uuid` or numeric; other key types will need additional cast branches in `migrate_table`.
- Sequences attached to identity columns in the target schema must be able to allocate fresh ids — running this against a populated target works as long as identity allocation does not collide.
- The migration is not transactional across tables by default; wrap your call in `BEGIN; ... COMMIT;` if you want all-or-nothing semantics.

## Recovery

`temp_migration` is the source of truth for what got migrated. To roll back manually:

```sql
-- Find rows inserted from a given source
SELECT new_id FROM public.temp_migration
WHERE source_schema = 'legacy' AND source_table = 'users';

-- Clear the mapping table to start over
TRUNCATE public.temp_migration;
```
